# 프로젝트: InfiniteTap (배터리리스 적응형 스위치 허브)

이 문서는 AI 코딩 에이전트가 현재 InfiniteTap 프로젝트의 진행 상황, 하드웨어 사양, 그리고 향후 개발 단계를 정확히 이해하고 작업을 이어받을 수 있도록 작성된 인수인계 컨텍스트입니다.

## 1. 프로젝트 개요
- **목표:** 블루투스 스위치 제어(Switch Control)를 지원하는 다양한 스마트 기기(Vision Pro, iPad, iPhone, Android 등)와 연동되는 배터리리스 초소형 BLE HID 스위치 허브 개발.
- **핵심 가치:** 기존 기성 보조공학 기기(예: AbleNet Blue2)의 높은 가격과 배터리 방전 문제를 해결하기 위해, USB-C를 통한 상시 유선 전원을 사용하는 지속 가능하고 경제적인 솔루션 구축.
- **폼 팩터:** 부피를 극도로 최소화하기 위해 전선이나 핀 헤더 없이 개발 보드와 소켓을 직접 납땜(Direct Solder)하는 동글 형태의 디자인.

## 2. 하드웨어 사양 및 핀 맵 (Pin Mapping)
- **MCU:** ESP32-C3 기반 초소형 개발 보드 (예: M5Stamp C3U / RISC-V 아키텍처, BLE 5.0 지원)
- **입력 단자:** 3.5mm 모노 소켓 (PJ-320) 1개
- **전원:** USB-C 포트를 통한 상시 버스 전원 공급 (배터리 없음, Deep Sleep 진입 불필요)
- **직결 배선 구조:**
  - PJ-320 뒤쪽 핀 (GND) ➡️ MCU `GND` 홀
  - PJ-320 앞쪽 핀 (Tip) ➡️ MCU `GPIO 8 (G8)` 홀
  *(주의: 합선 방지를 위해 PJ-320의 사용하지 않는 나머지 중간 핀들은 물리적으로 제거함.)*
- **내장 자원:**
  - **RGB LED (SK6812):** 내부적으로 `GPIO 2`에 연결됨
  - **물리 버튼:** 보드 중앙에 위치 (주황색 케이스 입력)

> 💡 **핀 변경 안내 (G0 ➡️ G8):**
> 기존에 사용하던 GPIO 0은 ESP32-C3의 스트래핑(Strapping) 핀으로, 부팅 시 스위치가 눌려 있으면 다운로드 모드로 진입하는 문제가 있었습니다. 이를 해결하기 위해 부팅에 영향을 주지 않는 GPIO 8로 배선을 변경하였습니다.

## 3. 현재 마일스톤 및 검증 상태
현재 전체 BLE 프로토콜을 구현하기 전, 물리적인 납땜 상태와 스위치 입력을 직관적으로 검증하기 위한 **'하드웨어 기본 동작 테스트 단계'**에 있습니다.

### 하드웨어 테스트 로직 (Pre-BLE Validation):
- **평상시 상태:** `GPIO 8`에 내부 풀업(Internal Pull-up)을 적용하여, 평소에는 내장 RGB LED(`GPIO 2`)가 켜져 있는 상태(흰색 또는 초록색)를 유지함.
- **입력 상태:** 외부 적응형 스위치를 누르면(G8이 GND와 쇼트되어 `LOW`가 되면), 내장 RGB LED가 즉시 **완전히 꺼짐(암전)**.
- **해제 상태:** 스위치에서 손을 떼면 LED가 다시 켜짐.

## 4. 향후 구현 과제 (다음 에이전트의 역할)
하드웨어 암전 테스트가 성공적으로 완료되면, 다음 에이전트는 실전 배포용 BLE HID 펌웨어를 구현해야 하며 요구 사항은 다음과 같습니다:

1. **BLE HID 키보드 프로필:** 표준 블루투스 키보드로 호스트에 인식될 것.
2. **타깃 키 전송:** 스위치 입력 시 호스트의 스위치 제어 환경이 인식할 수 있는 키값(기본값: `Space` 키 또는 `KEY_MEDIA_PLAY_PAUSE`, 변수화하여 커스텀 가능하도록 구성)을 전송할 것.
3. **지속적이고 빠른 자동 재연결(Auto-Reconnect):**
   - 호스트와의 연결이 끊어지면 즉시 재광고(Advertising)를 시작해야 함.
    - 호스트 기기가 잠자기에서 깨어나거나 켜질 때 1초 이내에 자동으로 재연결되어야 함.
4. **비차단형(Non-blocking) 아키텍처:**
   - BLE 연결 안정성을 위해 `delay()` 사용을 엄격히 금지하고, `millis()` 기반의 디바운스(Debounce) 로직을 적용할 것.
   - 입력 지연을 최소화하기 위해 BLE Connection Interval을 최적화할 것.
5. **내장 자원을 활용한 UI 상태 표시:**
   - **파란색 깜빡임:** BLE 광고 중 (페어링 대기 상태)
   - **초록색 켜짐:** 호스트와 BLE 연결 완료 상태
   - **빨간색 깜빡임:** 스위치 클릭 이벤트 발생 순간
   - **중앙 버튼 3초 이상 롱 프레스:** 기존 블루투스 본딩(페어링 정보)을 강제 삭제하고 즉시 초기 페어링 모드로 진입.

---
**새로운 에이전트를 위한 지침:** 이 파일을 읽고 아키텍처를 분석한 뒤, 사용자에게 기본 LED 테스트 코드 작성을 시작할지 아니면 바로 본 펌웨어 구현으로 들어갈지 확인하십시오.
